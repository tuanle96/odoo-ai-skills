# Pilot: AI Odoo Safety Gate + Instance Dossier

A fixed-scope, ~2-week engagement to stand up **odoo-ai-skills** on one partner
repo (or one client's project) and prove it on real pull requests.

> **odoo-ai-skills gives Claude Code / Codex fast local Odoo instance truth and a
> CI-bound evidence gate, so AI-written Odoo 17–19 changes are inspected,
> runtime-verified, and reported before PR, UAT, or release.**

The suite is open (LGPL-3) and the tools are free. This pilot is paid
enablement — you pay for the standing-up, the tuning to your instance and
vertical, and the team walkthrough, not for the software. See
[partner-enablement.md](partner-enablement.md) for the full service catalog.

---

## Goals

By the end of the pilot the partner team can, on their own instance:

1. Generate an **Instance Dossier** (what's really installed, what's customised,
   where the risk is) from a staging DB.
2. Run the **CI-bound evidence gate** on a real repo so AI-written changes are
   inspected, runtime-verified, and reported before merge — `Inspect → Verify →
   Gate → Report`.
3. Reproduce a failed gate locally, read the evidence, and fix the change —
   without help.

This is a **trust boundary**, not autopilot. The realistic target is
*agent-written, tool-verified, human-approved* — the gate hardens review, it
does not remove the human.

## What's included (week-by-week)

**Week 1 — Inspect + advisory gate**

- *Day 1–2 — Environment + doctor.* Install the suite (`npx skills add
  tuanle96/odoo-ai-skills`), confirm shell access to a **staging** DB, run the
  introspection engine against it, and shake out env issues. Build the local
  docs index if wanted.
- *Day 1–2 — Instance Dossier on staging.* Capture the live registry: installed
  modules, edition, Studio/OCA/local patches, entrypoint surface (`surface` /
  `esg`), effective per-role security on the high-risk models. Delivered as
  self-contained HTML + JSON.
- *Day 3–5 — CI gate on one repo, advisory mode.* Wire the gate into the
  partner's GitHub/GitLab CI on **one** repo. It runs but does not block —
  `deploy-gate` reports `approve | needs-human | block` as a comment/summary so
  the team sees the verdict without a hard stop yet.

**Week 2 — Verify + Gate (strict) + walkthrough**

- *Strict mode on high-risk models.* Turn the gate to blocking for a named set
  of high-risk models (the multi-company, accounting, stock, upgrade paths from
  the [high-risk playbooks](high-risk-playbooks.md)). Everything else stays
  advisory.
- *Team walkthrough.* One working session: how to read the Dossier, how to read
  an evidence bundle, how to reproduce a `block` locally, how the redaction
  rules work.
- *Evidence artifact review.* Walk a real (redacted) evidence bundle end to end
  and agree storage/retention per [artifact-governance.md](artifact-governance.md).

## Prerequisites

- **Staging DB access** with shell (`odoo-bin shell`) — self-hosted or an
  **Odoo.sh** branch. (No production credentials — see out-of-scope.)
- **GitHub or GitLab** for the target repo's CI.
- **Docker** available for the demo/e2e images.
- A named set of **high-risk models** the team cares about most.

If the client is on **Odoo Online (SaaS)**, the code gate does not apply — see
[odoo-online-advisory.md](odoo-online-advisory.md). The pilot then narrows to
the read-only advisory deliverables (Dossier, config audit, fit-gap).

## Out of scope

- **No production credentials** and no gating against production. Staging only.
- **No Odoo Online code gating** — impossible by construction (no custom code on
  that hosting). See [odoo-online-advisory.md](odoo-online-advisory.md).
- **No fixed-price custom development.** This pilot stands up the gate; feature
  work is a separate engagement.
- No private fork of the verifier. The tools stay the open tools — see
  [governance.md](governance.md).

## Success criteria

- The gate runs on **≥ 20 PRs** (real or synthetic) on the target repo.
- The team can **reproduce a failed gate locally** — pull the branch, run the
  evidence bundle, see the same `block`, and understand why.
- A Dossier can be regenerated by the partner without assistance.

## FAQ

**Is my data leaving the box?**
No. odoo-ai-skills is **local-first** — everything runs in your shell, no
account, no API key, no metadata leaves your environment. Evidence artifacts can
carry secrets/PII by design, so they are handled under
[artifact-governance.md](artifact-governance.md): redact before anything leaves
the machine, never commit raw dumps.

**What if we're on Odoo Online?**
Then there's no custom code to gate — the pilot becomes advisory-only. Read
[odoo-online-advisory.md](odoo-online-advisory.md) for exactly what works
(Dossier, config audit, fit-gap, migration-readiness) and what doesn't.

**Do you need production access?**
No. Staging DB access is enough, and it's safer. The whole point is to catch the
break before production.

**What does "CI-bound evidence gate" actually mean?**
CI produces signed evidence bound to the git diff — changed-line coverage,
runtime-path binding, scenario satisfaction — so "high coverage but runtime
breaks" becomes a `block`, not a green tick. It is a gate with an explicit trust
boundary, backed by human review. See the `odoo-testing` skill and the
[README](../README.md).
